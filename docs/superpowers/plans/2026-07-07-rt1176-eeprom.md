# RT1176 flash-emulated EEPROM Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement the AVR EEPROM C API in the RT1176 core (`cores/imxrt1176`), backed by flash emulation ported from `cores/teensy4/eeprom.c`, so the unmodified header-only `newdigate/EEPROM` library works — 4284-byte EEPROM emulated in the top 256 KB of the 16 MB FlexSPI1 NOR.

**Architecture:** A **core bring-up** (not a library move). Port `cores/teensy4/eeprom.c` verbatim-in-behavior — the sector-mapped wear-leveling + the RAM-resident FlexSPI program/erase primitives — retargeting RT1062→RT1176: FlexSPI1 base `0x400CC000`, `FLASH_BASEADDR 0x30FC0000`, `FLASH_SECTORS 63`. The core already runs all `.text` from ITCM (`imxrt1176.ld: *(.text*) > ITCM`), so the flash primitives run from RAM **automatically** — the self-programming-flash risk is architecturally handled, exactly as on Teensy 4. Not byte-identical → QEMU gate (qemu2 already models FlexSPI program/erase) + HW (persistence across reset) are the check.

**Tech Stack:** C (arm-none-eabi-gcc), i.MX RT1176 FlexSPI1 NOR (same IP as RT1062), `teensy-cmake-macros` build, QEMU `mimxrt1170-evk` via `evkb/tools/qrun`. Repos: core = `teensy-cores` (`cores/imxrt1176`), library = `newdigate/EEPROM` (`/Users/nicholasnewdigate/Development/EEPROM`, **unchanged, only build-linked**), gates = evkb (local), qemu2 (a small flash-init tweak). Commit to `master` each; **push only when the user asks** (Task 7).

- [ ] **Step 3: Write the gate firmware `evkb/eeprom_test/eeprom_test.cpp`**

```cpp
#include "Arduino.h"
#include "HardwareSerial.h"
#include <EEPROM.h>

struct Settings { uint32_t magic; int16_t a; int16_t b; char tag[6]; };

void setup() {
	Serial1.begin(115200);
	while (!Serial1) {}

	// --- Stage RW: byte-level write/read across the address space + a struct ---
	bool rw_ok = true;
	for (int i = 0; i < 256; i++) {
		uint8_t v = (uint8_t)(i * 7 + 3);
		EEPROM.write(i, v);
	}
	for (int i = 0; i < 256; i++) {
		if (EEPROM.read(i) != (uint8_t)(i * 7 + 3)) { rw_ok = false; break; }
	}
	Settings s = { 0xEE9702AA, -1234, 4321, {'R','T','1','1','7','\0'} };
	EEPROM.put(1000, s);
	Settings r;
	EEPROM.get(1000, r);
	if (r.magic != s.magic || r.a != s.a || r.b != s.b || strcmp(r.tag, s.tag) != 0) rw_ok = false;
	Serial1.print("EEPROM_RW="); Serial1.println(rw_ok ? "PASS" : "FAIL");

	// --- Stage WEAR: hammer one address past a full sector to force compaction+erase ---
	// A sector holds 2048 two-byte entries; >2048 writes to the same address
	// forces eepromemu_flash_erase_sector + rewrite. Verify the value + a
	// neighbor survive the compaction.
	const int A = 42, B = 43;
	EEPROM.write(B, 0x5A);
	bool wear_ok = true;
	for (int n = 0; n < 2100; n++) {
		EEPROM.write(A, (uint8_t)n);       // changes each time -> new entry each time
		if (EEPROM.read(A) != (uint8_t)n) { wear_ok = false; break; }
	}
	if (EEPROM.read(B) != 0x5A) wear_ok = false;   // neighbor survived the erase/compaction
	Serial1.print("EEPROM_WEAR="); Serial1.println(wear_ok ? "PASS" : "FAIL");

	Serial1.print("EEPROM_LENGTH="); Serial1.println(EEPROM.length());   // expect 4284
	Serial1.print("EEPROM_ALL="); Serial1.println((rw_ok && wear_ok) ? "PASS" : "FAIL");
}
void loop() {}
```
> `EEPROM.write` (the library) does read-before-write (`update` semantics via `EERef::operator=` → `eeprom_write_byte`, which skips if unchanged). The WEAR loop writes a *changing* value each iteration so every write appends an entry, filling the sector to force the erase path. Address `A=42` and `B=43` land in the same sector (`(addr>>2)%63`: 42>>2=10, 43>>2=10 → both sector 10) so the neighbor test proves the compaction preserved other data.

## Task 6: Hardware verification — persistence across reset (PAUSE for user)

The real EEPROM proof: data survives a power/reset cycle (a within-run RAM buffer would pass QEMU but fail this). Controller drives flash + VCOM + reset; the user confirms. Per [[rt1170-evkb-flashing]]: LinkServer, VCOM @115200; capture across the flash+reset window.

- [ ] **Step 1: Flash + first-boot capture**

Flash `eeprom_test/build/eeprom_test.elf` via `LinkServer flash MIMXRT1176:MIMXRT1170-EVKB load <elf>`, capturing the debug VCOM (`/dev/cu.usbmodem…` MCU-Link port) @115200. Expected on VCOM: `EEPROM_RW=PASS`, `EEPROM_WEAR=PASS`, `EEPROM_LENGTH=4284`, `EEPROM_ALL=PASS` — the emulation works on silicon (self-programming the real NOR).

> ⚠️ Self-programming risk watch: a hard-fault here = a flash-resident function on the flash-op path (recheck `nm` ITCM). Worst case (corrupted image) = recoverable by re-flashing. The EEPROM region is 15+ MB from the boot image.

- [ ] **Step 2: Persistence — reset WITHOUT re-flashing, read back**

The `eeprom_test` firmware re-runs `setup()` on every reset and re-writes, so for the *persistence* proof, use a second tiny firmware (or add a mode): read the struct at 1000 written in Step 1 and print it, **without writing first**. Simplest: build `evkb/eeprom_test/eeprom_persist_check.cpp` (a read-only sketch: `EEPROM.get(1000, r); Serial1.print("PERSIST magic="); Serial1.println(r.magic, HEX);`), flash it (which resets but does NOT touch the EEPROM region — a new image in low flash, EEPROM region untouched), and confirm VCOM prints `PERSIST magic=EE9702AA` — the struct written by the *previous* firmware survived the re-flash + reset. **This is the decisive persistence proof.** PAUSE — present the two VCOM captures to the user for confirmation.

## Notes on ordering & safety

- Tasks 1–3 are additive (shims, register defs, linker ASSERT) — the core keeps building. Task 4 adds `eeprom.c` (compiles standalone but isn't linked into a gate until Task 5). Task 5 is the first link+run — build errors point back to Tasks 2/4.
- **The self-programming-flash safety is architectural, not per-function:** the RT1176 core links `*(.text*)` into ITCM, so `eeprom.c`'s primitives + their callees (`memcpy`, `flash_wait`, `arm_dcache_delete`) all run from RAM automatically — same as Teensy 4, which marks nothing `FASTRUN` either. The Task-4/Task-5 `nm` check (symbols in `0x0..0x3FFFF`) confirms it. IRQs are disabled across each flash op (the ported `__disable_irq`/`__enable_irq`), covering the flash-resident vector table.
- qemu2 models FlexSPI program/erase already; the only qemu2 change is the `0xFF` erased-NOR init (Task 5 Step 1) — without it the zero-filled flash breaks the `0xFFFF` empty-slot scan.
