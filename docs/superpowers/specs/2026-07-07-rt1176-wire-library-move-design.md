# RT1176 Wire ‚Üí library move (full Teensy API) ‚Äî Design

**Status:** approved (design), ready for implementation plan
**Date:** 2026-07-07

## Goal

Move Wire (I¬≤C) out of the core (`cores/imxrt1176/Wire.{h,cpp}` + `Wire_instances.cpp`, teensy-cores) and into the **`newdigate/Wire`** library (`/Users/nicholasnewdigate/Development/Wire`, github), inline with how Teensy structures it (Wire is an Arduino **library**, split per peripheral family ‚Äî not a core file; same boundary as the just-moved WM8962 and SPI). Like the SPI move (and unlike the byte-identical WM8962 move), the RT1176 Wire is **re-implemented to the full Teensy Wire API** ‚Äî `TwoWire : public Stream`, all `requestFrom`/`beginTransmission`/`endTransmission` overloads, `setSDA`/`setSCL`, `int` overloads ‚Äî so standard Arduino/Teensy I¬≤C libraries drop in. The RT1176 code lands in a **dedicated file `WireIMXRT1176.{h,cpp}`** dispatched from `Wire.h`, matching Wire's per-peripheral-family file convention (`WireIMXRT` for Teensy-4, `WireKinetis` for Teensy-3). The Wire tests move into the Wire repo.

## Why this shape (exploration findings)

- The core's Wire is a **lean from-scratch** driver: `cores/imxrt1176/Wire.{h,cpp}` (a standalone `class TwoWire`, **not** `Stream`-derived, `BUFFER_LENGTH` 32) + `Wire_instances.cpp` (the `lpi2c{1,2,5}_hw` `hardware_t` tables of flat LPI2C register refs). It is a **polled master** (the `endTransmission`/`requestFrom` engine judges ACK/NACK at STOP via `MSR.NDF`, not early `TDF`) + an **interrupt-driven slave** whose ISR runs from `.fastrun` (ITCM) so it refills `STDR`/drains `SRDR` inside the `SCFGR2.CLKHOLD` clock-stretch window. Both master and slave are **HW-verified** (master to a real slave; slave 40/40 clean after a bench ground/pull-up fix ‚Äî the firmware was correct throughout). Nothing else in the core `#include`s Wire (only `Wire.cpp`/`Wire_instances.cpp` do), so the move creates no core‚Üílibrary coupling.
- `newdigate/Wire` is the **full Teensy Wire library** ‚Äî `Wire.h` dispatches by platform to `WireIMXRT.{h,cpp}` (`__IMXRT1052__`/`__IMXRT1062__`), `WireKinetis.{h,cpp}` (`__arm__ && TEENSYDUINO`), or an inline AVR `TwoWire`. Its latest commit already maps `Wire`/`Wire1`/`Wire2` to the **MIMXRT1060-EVKB** I¬≤C pins. The Teensy-4 `WireIMXRT` `TwoWire : public Stream` uses a `constexpr TwoWire(uintptr_t portAddr, const I2C_Hardware_t&)` with an `I2C_Hardware_t` (per-bus `pin_info_t sda_pins[2]/scl_pins[2]`, `clock_gate_register/mask`, `irq_number/irq_function`) accessed through a `port()` register overlay. **Our core is not `TEENSYDUINO`.**
- RT1176 LPI2C is the **same IP family** as the 1062, so the register contract (MCR/MSR/MCFGR/MCCR/MTDR/MRDR + slave SCR/SSR/SIER/SCFGR/SAMR/STDR/SRDR) is identical; only the platform layer differs (CCM clock roots + LPCG gating vs 1062's CCM_CSCDR/CCGR, IOMUXC pads/select-inputs, base addresses `LPI2C1@0x40104000`/`LPI2C2@0x40108000`/`LPI2C5@0x40114000`‚Ä¶ ‚Äî LPI2C5 is `0x40C34000` in the LPSR domain). **Reusing the Teensy-4 `WireIMXRT` engine wholesale on RT1176 is explicitly rejected:** our core Wire deliberately diverged to solve real silicon issues (the TDF-vs-NDF address-NACK race; the slave `CLKHOLD`/`TXDSTALL` + fastrun-ITCM ISR requirements). Adopting the Teensy-4 engine risks re-introducing bugs we already fixed and HW-verified. So ‚Äî exactly as for SPI ‚Äî the hybrid keeps **our proven register/clock/pin/slave logic** under the **Teensy API shape**.
- The core has **flat** `LPI2C1_*` register `#define`s, not an `IMXRT_LPI2C_t` port struct. Like SPI's `IMXRT_LPSPI_t`, the port struct is added to the core (register structs are core material in Teensy) and the driver moves out.
- The core **provides `Stream.h` and `Print.h`**, so adopting `TwoWire : public Stream` (the full Teensy API) is feasible without pulling anything else across.

## Scope

**In scope:** the full Teensy `Wire` API for **`Wire` (LPI2C1, Arduino header), `Wire1` (LPI2C2, QEMU-loopback slave persona), `Wire2` (LPI2C5, onboard eCompass/WM8962 codec)** ‚Äî **master and slave** (both HW-verified, both preserved) ‚Äî as a dedicated `WireIMXRT1176.{h,cpp}` in `newdigate/Wire`, dispatched from `Wire.h`. The core gains `IMXRT_LPI2C_t` + `IMXRT_LPI2C1/2/5_ADDRESS`; the core `Wire.{h,cpp}` + `Wire_instances.cpp` are deleted. The three Wire gates move into `newdigate/Wire/tests/`; the audio + SSD1306 gates that consume Wire stay in evkb and add a Wire-library import. HW-verified.

**Explicitly deferred (YAGNI):** I¬≤C DMA (our engine is polled-master/interrupt-slave; the Teensy-4 Wire path is also non-DMA ‚Äî no consumer needs it). Any 4th bus / `Wire3`. Adopting Teensy-4's 136-byte `BUFFER_LENGTH` (keep the classic Arduino 32 ‚Äî the WM8962 writes 4-byte registers, the SSD1306 pages ‚â§16 bytes). Applying the same "tests-with-library" move to the audio gates (they legitimately keep a Wire dependency through the codec).

## Architecture ‚Äî components

### Core (`teensy-cores`) ‚Äî register defs only
Add to `cores/imxrt1176/imxrt1176.h` (and the generator `tools/gen_imxrt1176_h.py`, since the header is auto-generated): an `IMXRT_LPI2C_t` struct (the LPI2C register-block overlay ‚Äî master regs `MCR@0x10 ‚Ä¶ MRDR@0x70` and the slave block `SCR@0x110 ‚Ä¶ SRDR@0x170`, matching the Teensy `imxrt.h` `IMXRT_LPI2C_t` layout with the correct reserved-word padding) + `#define IMXRT_LPI2C1_ADDRESS 0x40104000`, `IMXRT_LPI2C2_ADDRESS 0x40108000`, `IMXRT_LPI2C5_ADDRESS 0x40C34000`. Cross-check every offset against the flat `LPI2C1_*` addresses already in `imxrt1176.h` (`MCR@+0x10`, `MSR@+0x14`, `MIER@+0x18`, `MCFGR1@+0x24`, `MCCR0@+0x48`, `MFCR@+0x58`, `MTDR@+0x60`, `MRDR@+0x70`, `SCR@+0x110`, `SSR@+0x114`, `SIER@+0x118`, `SCFGR1@+0x124`, `SCFGR2@+0x128`, `SAMR@+0x140`, `SASR@+0x150`, `STDR@+0x160`, `SRDR@+0x170`). Additive (the core Wire still builds against it until Task N deletes it). Delete `Wire.{h,cpp}` + `Wire_instances.cpp` in the integration task.

### Library (`newdigate/Wire`) ‚Äî the dedicated `WireIMXRT1176.{h,cpp}`
- **Dispatch (`Wire.h`):** add `#elif defined(__IMXRT1176__)` ‚Üí `#include "WireIMXRT1176.h"` to the top platform chain (before the `__arm__ && TEENSYDUINO` ‚Üí `WireKinetis.h` arm), so RT1176 (which is **not** `TEENSYDUINO`) selects the new file.
- **`WireIMXRT1176.h`:** `class TwoWire : public Stream` with the **full Teensy API surface** copied in shape from the `WireIMXRT.h` declaration ‚Äî `begin()`/`begin(uint8_t)`/`begin(int)`/`end()`/`setClock`/`setSDA`/`setSCL`/`beginTransmission`(`uint8_t`+`int`)/`endTransmission`(`uint8_t`+`void`)/all five `requestFrom` overloads/`write`(`uint8_t`,`const uint8_t*`,size_t + the `int/long/unsigned` inline forms + `using Print::write`)/`available`/`read`/`peek`/`flush`/`onReceive`/`onRequest` ‚Äî plus an `I2C_Hardware_t`-style hardware-description struct (retargeted from `SPI_Hardware_t`'s RT1176 shape: `volatile uint32_t &lpcg; volatile uint32_t &clock_root; uint32_t clock_root_val;` + `pin_info_t`-style `sda`/`scl` mux/pad/`select_input`/`select_val`, `pad_ctl_val`, `IRQ_NUMBER_t irq`, `void (*irq_function)(void)`, `irq_priority`), a `port()` accessor over `IMXRT_LPI2C_t` (from the constructor's port address), and the **slave state** our engine needs (`is_slave`, `on_receive`/`on_request`, `s_rx_*`/`s_tx_*` buffers, `handle_slave_isr()`). `extern TwoWire Wire, Wire1, Wire2;`.
- **`WireIMXRT1176.cpp`:** method bodies **adapted from our core `Wire.cpp` verbatim in behavior** (flat `hw->mcr` refs ‚Üí `port().MCR` overlay fields; the free `hardware_t` ‚Üí the class's `I2C_Hardware_t`), preserving exactly: `begin()`/`begin(addr)` (LPCG ungate + clock root + IOMUXC mux/pad/select + master reset/enable, or slave `SCR/SAMR/SCFGR1/SCFGR2/SIER` + `attachInterruptVector`), `setClock` (the 24 MHz MCCR prescale/CLKLO/CLKHI math), `wait_flag`, `write`√óforms, `bus_recover`, `endTransmission` (START+addr, per-byte TXD, STOP, **judge NACK at STOP via NDF**), `requestFrom` (START+addr(R), RXD N-1, drain RDF, STOP), `read`/`peek`/`available` (slave-vs-master branch), and the fastrun `handle_slave_isr` (AVF/RDF/TDF/SDF + BEF/FEF W1C recovery). The API-only additions (`setSDA`/`setSCL` pin-select, `flush`, the `int`/`bool` overloads, `Stream` virtuals) wrap this engine. Add the `lpi2c{1,2,5}_hardware` tables with the **exact** values from the current `Wire_instances.cpp` (LPI2C1: `GPIO_AD_08`/`GPIO_AD_09`, mux `0x11` = ALT1|SION, pad `0x1E`, CLOCK_ROOT37/LPCG98, `IRQ_LPI2C1`; LPI2C2: CLOCK_ROOT38/LPCG99, `IRQ_LPI2C2`; LPI2C5: `GPIO_LPSR_05`/`GPIO_LPSR_04`, mux `0x10` = ALT0|SION, pad `0x0A`, `clock_root_val` `(1<<8)`, CLOCK_ROOT41/LPCG102, `IRQ_LPI2C5`) + `TwoWire Wire(IMXRT_LPI2C1_ADDRESS, lpi2c1_hardware)` etc. + the fastrun `wire{,1,2}_isr` trampolines. File-local `MSR_*`/`MCR_*`/`SCR_*`/`SSR_*`/`SIER_*`/`TX_CMD` macros carried from the core `Wire.cpp`.

## Data flow / behavior

I¬≤C API behavior is preserved ‚Äî master polled transfers (ACK/NACK judged at STOP), interrupt-driven slave with clock-stretch refill from ITCM ‚Äî now expressed through the Teensy `TwoWire : public Stream` surface. **Not byte-identical** (re-implementation on a different class structure) ‚Üí behavior validated on hardware, not by diff. `Wire2` remains the codec bus, so the WM8962 (and any future I¬≤C codec/sensor) keeps working unchanged.

## Testing

**QEMU gates** (now in `newdigate/Wire/tests/`, via `qrun`): `wire_master_test` (AT24C EEPROM on `lpi2c[0]` ‚Äî write/read-back + the multi-address bus scan that catches the TDF/NDF NACK regression), `wire_slave_test` (LPI2C2‚ÜíLPI2C1 loopback persona ‚Äî `onReceive`, address ACK, single- and multi-byte reads via the slave ISR), `wire_oled_test` (SSD1306 build/stream). These prove the register/engine plumbing regardless of real timing. The 3 audio gates' existing QEMU gates re-run to confirm the codec (`Wire2`) still enumerates after the include-path change.

**Hardware (final arbiter ‚Äî re-implementation, not byte-identical):**
- `wire_master_test` ‚Äî master writes/reads a real I¬≤C slave (per the LPI2C bring-up).
- `wire_slave_test` ‚Äî slave mode: `onReceive`/`onRequest`, multi-byte reads clean.
- `wire_oled_test` ‚Äî a real SSD1306 OLED renders (exercises the full write path end to end).
- The retarget surface (clock root/LPCG, pad SION/ODE/DSE, select-input daisy, IRQ) is the prime suspect for any regression ‚Äî cross-check every value against the HW-verified `Wire_instances.cpp`.

With the core Wire deleted, exactly one `TwoWire` exists in any build (the library branch) ‚Äî a stale duplicate would hard-fail the link.

## Risks

- **Re-implementation, not a move** ‚Äî the biggest risk, same as SPI. Mitigated by carrying our *proven* register/clock/pin/slave logic verbatim in behavior (only the access idiom changes: flat refs ‚Üí `port()` overlay), wrapping it in the Teensy API, and re-verifying on HW via all three gates (master, slave, OLED). The clock/pin/IRQ retarget is the error-prone surface ‚Äî cross-check every value against the current HW-verified `Wire_instances.cpp` + [[rt1176-lpi2c-wire]].
- **`IMXRT_LPI2C_t` struct correctness** ‚Äî field offsets must match the RT1176 LPI2C map, especially the **gap between the master block (ends ~`MRDR@0x70`) and the slave block (`SCR@0x110`)**; get the reserved padding right or every slave register read/writes the wrong address. Cross-check against the flat `LPI2C1_*` offsets and the Teensy `imxrt.h` `IMXRT_LPI2C_t`.
- **Bigger consumer blast radius than SPI** ‚Äî Wire lived in `cores/`, so it compiled into every gate for free. After the move, *every* Wire consumer needs an explicit import: the 3 Wire gates (move) **and** the 3 audio gates + 2 SSD1306 consumers (stay + import). Missing one surfaces as an `undefined reference to Wire`/`TwoWire::‚Ä¶` link error ‚Äî enumerate them (this spec's Testing section is the checklist) and build each after the move.
- **`Stream`-derived + `TEENSYDUINO` gate** ‚Äî the new file must compile without `TEENSYDUINO`; the RT1176 arm in `Wire.h` sits before the `__arm__ && TEENSYDUINO` arm, and `WireIMXRT1176.h` includes only the core's `Stream.h`/`Print.h`/`imxrt1176.h`/`core_pins.h` (no 1062-only Teensy dependency). Verify the dormant branch with a standalone `arm-none-eabi-g++ -c -D__IMXRT1176__` compile (core include path) before the integration task links it for real.
- **Test-harness paths** ‚Äî the moved gates reference the core/`teensy-cmake-macros`/`qrun` by workspace path (external deps); acceptable, documented in each test's CMake, matching the SPI precedent.
