# RT1176 PXP (Pixel Pipeline) library — Phase 1 design

**Date:** 2026-07-22
**Goal:** bring up the i.MX RT1176 PXP 2D graphics accelerator as a new
Teensy-style Arduino library (`newdigate/PXP`, MIT), starting with a **headless
memory-to-memory 2D engine**: fill, BitBlit, rotation and flips, with blocking
and async completion. Phase 1 of a five-phase roadmap that covers the whole of
RM chapter 52.

## Decisions (from brainstorm 2026-07-22)

| Question | Decision |
|---|---|
| v1 target | **Headless 2D engine** — PXP as a mem→mem blitter/compositor. Destination pixels are read back and asserted by the CPU. No display hardware involved. |
| Capability ceiling | **The whole chapter** — BitBlit, composition, Porter-Duff, rotation, resize, CSC, RGB+YUV formats — delivered in **five phases**. |
| Phasing | **5 phases, QEMU in lockstep.** Each phase extends `imxrt_pxp.c` only as far as that phase's features, then firmware, then HW verification. The two-gate rule holds throughout. |
| API shape | **Surface descriptors + fluent operation.** `PXPSurface` maps 1:1 onto the PS/AS/OUT register groups; `PXP.op().source(s).output(d).rotate(r).run()`. Later phases only *add* methods. |
| Completion | **Blocking + async twin** — `run()` polls `STAT[IRQ]`; `runAsync()` returns immediately with `EventResponder` on IRQ 57. Mirrors the SPI library's established idiom. |
| Phase 1 contents | Foundation + BitBlit **+ rotation/flips folded forward** (user decision): the QEMU model already does 180°/HFLIP/VFLIP, so only 90°/270° is new emulator work. |

## Phase roadmap

| Phase | Firmware | QEMU work |
|---|---|---|
| **1 · Foundation + BitBlit + geometry** | registers, clock/reset, library + manifest, surfaces, `fill`/`blit`, all 4 rotations + H/V flips, blocking + async, error surface, reachability probe | reset values, real bpp, `PS_BACKGROUND` fill, 90°/270° |
| **2 · Formats & framing** | full pixel-format matrix, clipping, letterboxing, in-place processing | format decode table |
| **3 · Compositing** | AS engine, global/local alpha, ROP, colour key, Porter-Duff | AS fetch + blend math |
| **4 · Scaling** | `PS_SCALE`/`PS_OFFSET`, decimation + bilinear | the scaler |
| **5 · YUV / CSC** | `CSC1` coefficients, YUV444/422/420/400, planar `UBUF`/`VBUF` | CSC + planar fetch |

## Established facts (grounding)

Verified during the brainstorm against the RM (`rt1170/rm_full.txt`, ch. 52),
the core, and the QEMU fork.

- **Block:** base `0x4081_4000`, **IRQ 57**, clock `clk_enable_pxp` =
  **LPCG127** off `BUS_CLK_ROOT` (RM `:83430`, `:85832`). PXP also drives
  **eDMA event 33** (RM `:4477`). IRQ 57 appears in **both** the CM7 and CM4
  interrupt tables (RM `:2972`, `:3733`) — a CM4-owned PXP is therefore
  plausible later, but is not in scope here.
- **This is the "lite" PXP:** ~35 registers, `0x000`–`0x440`. There is **no**
  LUT, dither, histogram or WFE block. (`STAT` documents a
  `LUT_DMA_LOAD_DONE_IRQ` bit at RM `:186396` with no corresponding LUT
  registers in the memory map — treat as a doc artefact carried over from
  i.MX6, not a feature.)
- **Reset values matter:** `CTRL` resets to `C000_0000h` — **`SFTRST` (bit 31)
  and `CLKGATE` (bit 30) are both set**, i.e. PXP is held dead until the
  documented sequence runs. `PS_SCALE` resets to `1000_1000h` (unity),
  `PS_CLRKEYLOW`/`AS_CLRKEYLOW` to `00FF_FFFFh`, and `CSC1_COEF0..2` to
  `0400_0000h`/`0123_0208h`/`079B_076Ch`.
- **RM §52.5 initialization:** set `CTRL[SFTRST]`; then clear `CTRL[SFTRST]`
  **and** `CTRL[CLKGATE]`; configure registers; set `CTRL[ENABLE]=1`; poll
  `STAT`. Repeat configure→enable→poll per operation.
- **CTRL bit layout** (confirmed, `MIMXRT1176_cm7.h` via the QEMU header):
  `ENABLE`(0), `IRQ_ENABLE`(1), `ROTATE`[9:8], `HFLIP`(10), `VFLIP`(11),
  `CLKGATE`(30), `SFTRST`(31). `STAT[IRQ]` = bit 0. Coordinate registers pack
  X in [29:16] and Y in [13:0].
- **FORMAT is three separate namespaces, not one** (transcribed from RM
  §52.6.3 and §52.6.12 during the Task 1 review): `PS_CTRL[FORMAT]` is bits
  [5:0] and **has no `0x00`** encoding; `OUT_CTRL[FORMAT]` is bits [4:0] and
  **has no `0x24`**; `AS_CTRL[FORMAT]` is bits [7:4], only 4 bits wide. The
  same format therefore has *different* encodings by role — `RGB565` is `0x0E`
  in both PS and OUT, but a 32-bit ARGB surface is `0x04` as a source and
  `0x00` as a destination. See the amendment below.
- **Output composition (RM §52.3.1.23):** the output rectangle is composed from
  the PS buffer *or* the `PS_BACKGROUND` register according to the programmed
  PS coordinates. This is what makes a solid `fill` a real hardware operation.
- **Known hardware limitations (RM §52.3.4.1):** (1) with the `NEXT` queue the
  interrupt-enable setting must be identical across queued frames; (2) 180°/270°
  are unsupported during LCD handshake; (3) **PS rotation and decimation cannot
  be used together**; (4) **no rotate combined with flip, scaling or decimation
  on an unaligned buffer**. (3) and (4) constrain Phases 1 and 4.
- **Error model (RM §52.3.4):** PXP does minimal validation of its control
  registers. It does monitor bus transactions and reports AXI read/write errors
  in `STAT`, aborting on error, but **does not report the failing address**.
- **D-cache is OFF in this core**, documented at
  `cores/imxrt1176/imxrt1176.h:849-854`, where `arm_dcache_delete` and
  `arm_dcache_flush_delete` are deliberate no-ops. Phase 1 therefore needs no
  cache maintenance — but PXP correctness *depends* on that fact, so it is
  recorded here rather than rediscovered later.
- **Memory map** (`imxrt1176.ld`): ITCM `0x0000_0000`+256K, DTCM
  `0x2000_0000`+256K, OCRAM `0x2024_0000`+512K, FLASH `0x3000_0000`+16M, SDRAM
  `0x8000_0000`+64M. `DMAMEM` places buffers in OCRAM; `extmem_malloc` in SDRAM.
- **Nothing PXP-related exists in firmware today.** `imxrt1176.h` has zero PXP
  definitions. The only prior artefacts are the RT1062 reference copy in
  `cores/teensy4/imxrt.h:7878-8040` (field layout largely reusable; base, IRQ
  and clock are all wrong for RT1176) and the QEMU model.

### The QEMU model, as it stands

`qemu2/hw/dma/imxrt_pxp.c` (220 lines, GPL-2.0-or-later) already implements a
synchronous PS→OUT rectangle copy honouring pitches and `ROTATE(180)`/`HFLIP`/
`VFLIP`, sets `STAT[IRQ]`, self-clears `ENABLE`, and implements the MXS
`SET/CLR/TOG` aliases. **Three divergences were identified during this
brainstorm and must be fixed as Phase-1 QEMU work:**

1. **Reset values are wrong.** `imxrt_pxp_reset_hold()` does
   `memset(s->regs, 0, ...)`, so `CTRL` reads `0000_0000h` instead of
   `C000_0000h`. Firmware that **skips the `SFTRST`/`CLKGATE` sequence entirely
   would pass in QEMU and be dead on silicon.** This is the single most
   dangerous divergence in the current model.
2. **`PS_BACKGROUND` is not implemented** — the model writes nothing for output
   pixels outside the PS rectangle, so `fill` does not work in emulation.
3. **`pxp_bpp()` is a two-way guess** (`format <= 0x5 ? 4 : 2`) rather than a
   real format decode, and `PXP_MAX_DIM` silently clamps any dimension above
   1024.

## 1. Placement — core registers, library driver

Following the convention already established by SPI, Wire and Audio:

- **Registers go in the core.** A `/* --- PXP --- */` block in
  `cores/imxrt1176/imxrt1176.h`: base, the ~35 registers with their
  `SET`/`CLR`/`TOG` aliases, the `CTRL`/`STAT`/`OUT_CTRL`/`PS_CTRL` bitfield
  macros, `IRQ_PXP = 57`, and the LPCG127 clock gate.
- **The driver goes in a new library repo,** `github.com/newdigate/PXP` (MIT,
  clean-room from RM ch. 52 — no vendor driver source is copied), registered as
  `_evkb_lib(PXP ${_dev}/PXP https://github.com/newdigate/PXP <sha> .)` and
  consumed with `import_evkb_library(PXP)`.
- `tools/license-audit.sh`: `REPOS +=` the PXP repo, `GATES +=` the new gate.

This deliberately avoids the `ST7735_t3` pattern (hardcoded `EVKB` path, raw
`import_arduino_library`), which predates the manifest.

## 2. API surface

```cpp
enum PXPRotation : uint8_t { PXP_ROT_0, PXP_ROT_90, PXP_ROT_180, PXP_ROT_270 };

// Abstract tokens, NOT hardware encodings -- the PXP uses a different
// encoding for the same format in each of PS_CTRL[5:0], OUT_CTRL[4:0] and
// AS_CTRL[7:4], and not every format is legal in every role.  Translate per
// role; PXP_FMT_NA means "not representable there".  Phases 2/5 extend this.
enum PXPFormat : uint8_t { PXP_ARGB8888 = 0, PXP_XRGB8888, PXP_RGB565 };

#define PXP_FMT_NA 0xFFu
uint8_t  pxpPsFormat(PXPFormat f);    // -> PS_CTRL[FORMAT]
uint8_t  pxpOutFormat(PXPFormat f);   // -> OUT_CTRL[FORMAT]
uint16_t pxpBitsPerPixel(PXPFormat f);

enum PXPError : uint8_t {
    PXP_OK = 0,
    PXP_ERR_BUSY,          // an operation is already running
    PXP_ERR_TIMEOUT,       // STAT[IRQ] never asserted
    PXP_ERR_CONFIG,        // geometry does not fit / bad surface
    PXP_ERR_UNREACHABLE,   // surface not visible to a bus master
    PXP_ERR_ALIGN,         // rotate/flip on an unaligned window (RM §52.3.4.1)
    PXP_ERR_AXI_READ,      // STAT AXI read error
    PXP_ERR_AXI_WRITE,     // STAT AXI write error
    PXP_ERR_NOT_BEGUN,     // begin() has not been called
    PXP_ERR_FORMAT,        // format not valid in that PS/OUT role
    PXP_ERR_UNIMPLEMENTED, // not yet implemented in this phase
};

struct PXPSurface {
    void      *data;
    uint16_t   width, height;
    uint16_t   pitch;        // bytes/row of the RGB/luma plane; 0 == INVALID
    PXPFormat  format;

    PXPSurface(void *d, uint16_t w, uint16_t h, PXPFormat f, uint16_t pitchBytes = 0);
    uint16_t bitsPerPixel() const;  // the primitive: sub-byte formats exist
    uint8_t bytesPerPixel() const;  // 0 when not a whole number of bytes
    size_t  sizeBytes() const;
    bool    reachable() const;   // bus-master address-range check (§5)
};

class PXPOp {
public:
    PXPOp &source(const PXPSurface &s);
    PXPOp &output(const PXPSurface &o);
    PXPOp &outputAt(uint16_t x, uint16_t y);  // destination placement
    // Deleted rvalue overloads: binding a temporary surface would dangle.
    PXPOp &source(const PXPSurface &&) = delete;
    PXPOp &output(const PXPSurface &&) = delete;
    PXPOp &background(uint32_t argb8888);     // PS_BACKGROUND
    PXPOp &rotate(PXPRotation r);
    PXPOp &flip(bool horizontal, bool vertical);
    PXPError run(uint32_t timeout_ms = 100);              // blocking
    PXPError runAsync(EventResponder *onComplete = nullptr);
};

class PXPClass {
public:
    bool     begin();                         // RM §52.5 reset + clock sequence
    void     end();
    PXPOp    op();                            // a fresh, zero-initialised op
    PXPError fill(const PXPSurface &dst, uint32_t argb8888);
    PXPError blit(const PXPSurface &src, const PXPSurface &dst);
    bool     busy() const;
    PXPError wait(uint32_t timeout_ms = 100);
    PXPError lastError() const;
};
extern PXPClass PXP;
```

`PXP.op()` returns a **fresh** operation rather than exposing persistent
builder state on the singleton. Configure-once-run-many would be marginally
more convenient, but stale configuration silently leaking from one blit into
the next is a genuinely nasty failure mode; a fresh op makes every operation
self-describing.

Later phases add only new methods — `.alpha()`, `.colorKey()`, `.porterDuff()`
(Phase 3), `.scaleTo()`, `.decimate()` (Phase 4) — plus new `PXPFormat`
enumerators (Phases 2 and 5). No Phase-1 signature changes.

## 3. Operation semantics

**`fill(dst, argb)`** programs `PS_BACKGROUND` and positions the PS rectangle
outside the output window, so every output pixel is composed from the
background register (RM §52.3.1.23). `OUT_LRC` spans the whole destination.

**`blit(src, dst)` and `.outputAt(x, y)` retarget the output window, they do not
use `OUT_PS_ULC` to offset within a larger buffer.** PXP always writes the entire
rectangle described by `OUT_LRC`, so offsetting via `OUT_PS_ULC` while leaving
`OUT_LRC` at full-framebuffer size would repaint everything outside the sprite
with `PS_BACKGROUND`. Instead the library sets:

- `OUT_BUF   = dst.data + y * dst.pitch + x * dst.bytesPerPixel()`
- `OUT_PITCH = dst.pitch` (the *framebuffer* pitch, so rows step correctly)
- `OUT_LRC`  = window dimensions − 1
- `OUT_PS_ULC = (0,0)`, `OUT_PS_LRC` = window dimensions − 1 (PS covers the
  whole window)

Surrounding pixels are then untouched by construction — which the gate asserts
explicitly.

**Geometry.** The destination window is `src.width × src.height` for
`PXP_ROT_0`/`PXP_ROT_180`, and **`src.height × src.width` for `PXP_ROT_90`/
`PXP_ROT_270`** (the axes swap). The library validates that the window fits
inside the output surface at `(x, y)` and returns `PXP_ERR_CONFIG` otherwise.
`HFLIP`/`VFLIP` compose with rotation; per RM §52.3.4.1(4) that combination
requires an aligned window, so a rotate or flip requested on a misaligned
`OUT_BUF` returns `PXP_ERR_ALIGN` rather than producing corrupt output. The
exact alignment requirement is transcribed from RM §52.3.1.17 (block size
selection) at plan time and confirmed on silicon.

## 4. Completion and errors

`run()` sets `CTRL[ENABLE]` and polls `STAT` to completion with a timeout,
returning `PXP_ERR_TIMEOUT` rather than hanging. `runAsync()` sets
`CTRL[IRQ_ENABLE]`, enables IRQ 57 in the NVIC, and returns immediately;
the ISR clears `STAT[IRQ]` (write-1-to-clear via `STAT_CLR`) and triggers the
supplied `EventResponder`. `busy()`/`wait()` cover the poll-for-async case.

`STAT`'s AXI read/write error bits are decoded into `PXP_ERR_AXI_READ`/
`PXP_ERR_AXI_WRITE` and latched in `lastError()`. Note the RM warns these do
not identify the failing address, so the library reports them faithfully rather
than pretending to localise the fault. **These two error paths are surfaced but
not gated in Phase 1** — neither QEMU nor a well-behaved board provides a bus
fault to inject. Error injection is noted as QEMU follow-up work.

## 5. Memory rules — bus-master reachability

PXP is an AXI bus master, so its surfaces must live in memory a master can
reach. OCRAM (`DMAMEM`), SDRAM (`extmem_malloc`) and FLASH (source only, being
read-only) are reachable. **CM7 TCM is the open question**: DTCM/ITCM are
expected to be invisible to other masters, but this project has already found
that the CM4's TCM *is* reachable through a system alias
(`0x2020_0000`), so the CM7 case must not be assumed.

Phase 1 therefore includes a **reachability probe**: the gate attempts a small
blit sourced from DTCM and from ITCM on real silicon and reports the result as
a token. The outcome is recorded in the library docs and *determines* what
`PXPSurface::reachable()` enforces — rather than the guard encoding a guess.

This matters because it is a **guaranteed QEMU divergence**: QEMU's
`dma_memory_read` will happily read DTCM and pass, whichever way silicon
behaves. Converting the question into an explicit, asserted token is what stops
it from becoming a silent hardware-only failure much later.

If the probe confirms TCM is unreachable, `reachable()` rejects
`[0x0000_0000, 0x0004_0000)` and `[0x2000_0000, 0x2004_0000)`, and an
unreachable surface fails fast with `PXP_ERR_UNREACHABLE`. An output surface in
FLASH is likewise rejected.

## 6. QEMU model work (Phase 1)

In `qemu2` (GPL-2.0-or-later; never linked into firmware — the licence firewall
is one-way):

1. **Correct reset values** in `imxrt_pxp_reset_hold()`: `CTRL = 0xC000_0000`,
   `PS_SCALE = 0x1000_1000`, `PS_CLRKEYLOW`/`AS_CLRKEYLOW = 0x00FF_FFFF`,
   `CSC1_COEF0..2` as documented. This closes the skip-the-reset-dance hole.
2. **`PS_BACKGROUND` composition** so `fill`, and output pixels outside the PS
   rectangle, behave as the RM describes.
3. **90°/270° rotation**, including the output-axis swap and its composition
   with `HFLIP`/`VFLIP`.
4. **Real per-format bpp decode** for the Phase-1 formats, replacing the
   two-way guess.

`PXP_MAX_DIM`'s silent 1024 clamp is left as-is for Phase 1 (all Phase-1
surfaces are far smaller) but is documented as a known model limitation.

## 7. Gate — `examples/display/pxp_blit_test` (CM7)

**The gate computes its own expected result in software on the device** and
compares it against what PXP produced, instead of asserting hard-coded
constants. Because the QEMU model is authored by this project, hard-coded
expectations risk enshrining a model bug as the specification; a
software reference blit cannot. Hard-coded values are used only for the
handful of tokens that must be identical across QEMU and HW transcripts.

Coverage. All but the last emit a `PXP_*=PASS` token plus a deterministic
checksum; the probe emits a value token instead, because its purpose is to
*establish* a fact rather than assert a known one:

- `PXP_BEGIN` — after `begin()`, `CTRL` reads back with `SFTRST` and `CLKGATE`
  clear (this is the assertion that the reset dance actually ran).
- `PXP_FILL` — solid fill, every pixel checked.
- `PXP_BLIT` — full-surface same-format copy vs the software reference.
- `PXP_SUBRECT` — offset blit; asserts both the placed pixels **and** that the
  surrounding framebuffer is byte-for-byte untouched (proves the output-window
  retargeting of §3).
- `PXP_ROT90`, `PXP_ROT180`, `PXP_ROT270`, `PXP_HFLIP`, `PXP_VFLIP` — each vs
  the software reference, including the 90°/270° axis swap.
- `PXP_ASYNC` — `runAsync()` + `EventResponder` callback fires; IRQ 57 path.
- `PXP_OCRAM`, `PXP_SDRAM` — the same blit with surfaces in `DMAMEM` and in
  `extmem_malloc` memory.
- `PXP_TCM=<dtcm>,<itcm>` — the reachability probe (§5). Emitted as an observed
  value, **not** a pass/fail: on QEMU it is expected to report both readable, and
  the silicon reading is the one that determines what `reachable()` enforces.
  A QEMU-vs-HW difference here is a documented finding, not a gate failure.
- `PXP_ALL=PASS` — overall (excludes `PXP_TCM`, which is an observation).

Two-gate rule: QEMU green, then HW-verified on the EVKB, with
`transcript_qemu.txt` and `transcript_hw_evkb.txt` committed and any divergence
documented rather than absorbed.

## 9. Risks

- **The QEMU model is authored by this project**, so it is not an independent
  oracle. Mitigated by the self-validating gate (§7) and by HW verification.
- ~~**Format encodings** for `PS_CTRL`/`OUT_CTRL` still need transcribing.~~
  **RESOLVED, and it was a real defect** — see the amendment below. This risk
  materialised exactly as written: the unconfirmed placeholder
  `PXP_ARGB8888 = 0x00` was a *reserved* value in `PS_CTRL`, and QEMU's bpp
  guess would have masked it into a green gate.
- **Alignment rules** for the rotate/flip combination (RM §52.3.4.1(4)) are
  stated qualitatively in the RM; the concrete requirement must be pinned down
  and confirmed on silicon.
- **TCM reachability** is genuinely unknown (§5) — hence a probe rather than an
  assumption.
- **D-cache dependency**: correctness currently relies on the D-cache being off.
  If a future build enables it, every PXP surface needs real maintenance. The
  library documents this so it fails loudly in review rather than silently in
  pixels.

## Amendment 1 (2026-07-22, during Task 1) — FORMAT is three namespaces

The original §2 asserted that `PXPFormat`'s enumerator values *were* the
hardware FORMAT encodings, with a single `0x3F` field mask. **That is wrong**,
and the Task 1 code-quality review caught it before any register write existed.
Verified against RM rev.5:

| Role | Register field | Mask | ARGB8888 | XRGB8888 | RGB565 |
|---|---|---|---|---|---|
| Source | `PS_CTRL[FORMAT]` [5:0] | `0x3F` | `0x04` | `0x04` | `0x0E` |
| Destination | `OUT_CTRL[FORMAT]` [4:0] | `0x1F` | `0x00` | `0x04` | `0x0E` |
| Alpha surface (Phase 3) | `AS_CTRL[FORMAT]` [7:4] | `0x0F` | `0x00` | `0x04` | `0x0E` |

The full `AS_CTRL[FORMAT]` set, transcribed from RM §52.6.22 for Phase 3:
`0x0` ARGB8888, `0x1` RGBA8888, `0x4` RGB888, `0x8` ARGB1555, `0x9` ARGB4444,
`0xA` RGBA5551, `0xB` RGBA4444, `0xC` RGB555, `0xD` RGB444, `0xE` RGB565.

`PS_CTRL` has **no `0x00`** encoding and `OUT_CTRL` has **no `0x24`**, so the
placeholder `PXP_ARGB8888 = 0x00` would have written a *reserved* value into
`PS_CTRL[FORMAT]`.

**Why this mattered more than an ordinary typo:** QEMU's `pxp_bpp()` is
`format <= 0x5 ? 4 : 2`, which returns the correct byte count for `0x00` for
the wrong reason. The gate would have gone green in emulation and failed only
on silicon in Task 11 — the precise failure mode the two-gate rule exists to
prevent, and a reminder that a self-authored emulator is not an oracle.

**Resolution:** `PXPFormat` became an abstract token enum with per-role
translators `pxpPsFormat()` / `pxpOutFormat()` returning `PXP_FMT_NA` when a
format has no encoding in that role (rejected as `PXP_ERR_FORMAT`). Each
translator is a `switch` with **no `default:`**, so adding a format without
teaching every table is a `-Wswitch` compile error rather than a silent wrong
value. `pxpBitsPerPixel()` replaced a two-way `?:` guess for the same reason —
and note the guess it replaced was the same anti-pattern §6 orders removed from
the QEMU model, which had quietly reappeared in the firmware.

Also folded in from that review: `at()` → `outputAt()` (Phase 3 needs AS
placement too, and renaming later would be an API break); deleted rvalue
overloads on `source()`/`output()`; overflow-safe `pitch` computed in 32 bits
with `0` meaning invalid; `PXP_ERR_NOT_BEGUN` / `PXP_ERR_FORMAT` /
`PXP_ERR_UNIMPLEMENTED` so stubs and setup errors stop masquerading as
`PXP_ERR_CONFIG`; and the `#error` guard replaced by the sibling libraries'
`#if defined(__IMXRT1176__)` wrapper, which does not pre-emptively lock out a
future CM4 client.

## Forward notes for Phases 2-5 (from the Task 5 review)

- **Stale register defaults across ops.** `_program()` relies on post-`begin()`
  reset defaults for the registers it does not write — `PS_SCALE` (unity),
  `PS/AS_CLRKEY` (keying off), `CSC1` (bypass), `AS_CTRL` (0). Correct for
  Phase 1's one-op-at-a-time same-format flow, but once the scaling/compositing/
  CSC phases start writing these, a value left by a prior op can leak into a
  later op that does not rewrite it. Those phases must either write every
  register they depend on each op, or reset the unused ones explicitly.
- **Same-format guard** (`_src->format != _dst->format → PXP_ERR_FORMAT`) is a
  Phase-1 contract; Phase 5 (CSC/YUV) is where cross-format becomes meaningful
  and relaxes it.
- **AS-disable is unverifiable in QEMU** — the model composites no alpha surface
  at all, so the degenerate-AS-rectangle park is HW-only. Confirm on silicon in
  Phase 3 that `AS_CTRL` at reset truly contributes nothing.

## Amendment 2 (2026-07-23, Task 11 HW verification) — four silicon findings

Flashing `pxp_blit_test` to the real EVKB turned up **two fundamental bugs the
QEMU model could not see**, plus two conservative guards that the silicon proved
wrong. Final state: `PXP_ALL=PASS` on hardware, and QEMU==HW on every functional
token and checksum (sole divergence: `PXP_TCM_STAT` `0x0` vs `0x30000`, the STAT
block-coordinate field QEMU does not model — an observation line).

1. **CSC1 colour-mangling (critical, firmware).** The PXP runs every PS source
   pixel through the CSC1 YUV→RGB colour matrix by default — `CSC1_COEF0[BYPASS]`
   (bit 30) resets to 0, and `CSC1_COEF0..2` reset to the default YUV
   coefficients. Every RGB blit came back colour-transformed. Diagnosed with a
   positional pixel dump (`src 0x1000 → dst 0x1082…`, deltas of the RGB565 green
   mask). Fixed: `_program()` sets `PXP_CSC1_BYPASS` for the RGB path (Phase 5
   YUV makes it conditional). **QEMU passed the whole time because it models no
   CSC1** — the textbook two-gate finding.

2. **Non-square 90/270 rotation geometry (firmware + QEMU model).** For
   output-stage rotation (ROT_POS=0), `OUT_LRC`/`OUT_PS_LRC` must be the
   **source** dimensions; the hardware rotates that source frame and lays the
   rotated result out via `OUT_PITCH`. §3's swap-into-`OUT_LRC` only worked
   because square surfaces have source dims == rotated dims. Cracked with a 4×2
   diagnostic + NXP's `fsl_pxp` reference (which sidesteps this by only rotating
   square surfaces). Firmware: `OUT_LRC`/`OUT_PS_LRC` = source dims, fit-check/
   placement = rotated extent, ROT_POS explicitly 0. QEMU model rewritten to
   forward-map the source frame; both gates now green with identical checksums.

3. **DTCM is bus-master reachable (guard relaxed).** The probe (§5) reads CM7
   DTCM cleanly with no AXI error — the "bus masters can't see TCM" premise in
   §4/§5 is **false** on this FlexRAM SoC. `reachable()` now accepts DTCM
   (`0x20000000`+256K); ITCM stays rejected (unprobed, code space). `PXP_GUARD`
   reports `dtcm:permits`, consistent with `PXP_TCM=readable`.

4. **Misaligned rotate-alone is fine (guard relaxed).** `PXP_ALIGN=off2:ok` — a
   2-byte-offset rotate is correct on silicon, matching RM §52.6.4 ("any byte
   alignment is valid for OUT_BUF"). The alignment guard now fires only for the
   one RM-documented hazard (§52.3.4.1(4)): rotate **combined with** flip on a
   <64B-aligned buffer. Rotate-alone and flip-alone run at any alignment.

**Confirmed correct on silicon:** the Amendment-1 `PS_BACKGROUND` RGB888→output
conversion (`PXP_FILL` matches QEMU), the CW rotation direction (ROT180/270
match), and every blit/subrect/async/SDRAM path.

**Process note:** NXP's BSD-3 `fsl_pxp.c` was *studied* to learn the hardware
rotation convention (a non-copyrightable fact the RM left ambiguous), not
copied — the clean-room firewall holds; no NXP or GPL source entered the MIT
library.
