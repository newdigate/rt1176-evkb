# RT1176 `tone()` / `noTone()` — Design

**Status:** approved (brainstorming) — ready for implementation plan
**Date:** 2026-07-04
**Target:** `cores/imxrt1176` Arduino core for the MIMXRT1176-EVKB. QEMU-gated (`-icount`) **and** hardware-verified (Saleae).

## Goal

Implement Arduino `tone(pin, frequency, duration=0)` / `noTone(pin)` — a single square-wave "tone" on any Arduino-header digital pin — mirroring the Teensy 4 reference (`cores/teensy4/tone.cpp`), which drives it from a shared `IntervalTimer`. Both functions are already declared in `cores/imxrt1176/WProgram.h:73-75`.

## Scope

**In:** `tone(pin, freq, duration)` + `noTone(pin)`; any mapped header pin; optional `duration` (auto-stop after N ms); frequency 1 Hz..65535 Hz; single tone at a time; pin driven via the GPIO atomic toggle register; backed by one `IntervalTimer` channel (shared with the PIT pool — leaves 3 for user timers).

**Out (YAGNI):** multiple simultaneous tones; Teensy's smooth same-pin frequency transition (we simply restart the timer at the new rate — negligible glitch); any PWM/`analogWrite` interaction beyond sharing a PIT channel via the IntervalTimer pool; non-header pads.

## Reference & hardware facts

- **Reference:** `cores/teensy4/tone.cpp` — `static IntervalTimer tone_timer` toggles the pin every half-period; `usec = 500000/freq`; `duration` counted in toggles; single-tone. We mirror it minus the mid-cycle frequency transition.
- **Substrate present:** `IntervalTimer` (PIT1, 4 shared channels) done + HW-verified (the just-completed peripheral). `tone/noTone` already declared. `pinMode`/`digitalRead` exist. GPIO atomic registers per port (`cores/imxrt1176/imxrt1176.h`, and documented in `digital.c`): `DR +0x00`, `GDIR +0x04`, `PSR +0x08`, `DR_SET +0x84`, `DR_CLEAR +0x88`, `DR_TOGGLE +0x8C`.
- **Pin access:** `digital_pin_to_info[pin]` gives `.gpio` (normal-port base, ALT5) and `.bit`. Toggle = `*(gpio+0x8C) = (1<<bit)`; clear = `*(gpio+0x88) = (1<<bit)`. Header pins are on GPIO2/3/5/6 (per the attachInterrupt rework).
- **Math:** pin toggles every `usec = 500000/freq` µs ⇒ full period `1/freq` ⇒ **pin frequency = `freq` Hz**. `duration>0` ⇒ toggles `= 2·freq·duration/1000`; total time `= toggles·usec = duration` ms.

## Architecture & files

```
cores/imxrt1176/
  tone.cpp   — tone()/noTone(), static state, tone_interrupt() ISR, local DR_TOGGLE/DR_CLEAR accessors
             — built on IntervalTimer; #include "Arduino.h" (pinMode, IntervalTimer, digital_pin_to_info)
evkb/
  tone_test/ — QEMU gate (-icount): edge-count freq, duration auto-stop, noTone; from interval_timer_test
```

- One file; no header changes (DR_TOGGLE/DR_CLEAR accessors are file-local, like Teensy's `TONE_*` macros). `WProgram.h` already declares the API. `tone.cpp` compiles into the core via the existing `*.cpp` glob.

## API & mechanism

**Static state:** `tone_pin` (255 = idle), `tone_base` (GPIO port base), `tone_mask` (`1<<bit`), `tone_toggle_count`, `static IntervalTimer tone_timer`.

- **`tone(uint8_t pin, uint16_t freq, uint32_t duration)`**:
  - `if (pin >= CORE_NUM_DIGITAL) return;` look up `p = &digital_pin_to_info[pin]`; `if (p->gpio == 0) return;` (unmapped).
  - `if (freq < 1) freq = 1;` `float usec = 500000.0f / freq;`
  - `duration>0` → `count = (uint32_t)((uint64_t)freq * duration / 1000) * 2;` (min 1; Teensy makes it odd for whole cycles — match in plan). `duration==0` → `count = 0` (infinite sentinel).
  - `__disable_irq();` if `pin != tone_pin` and a tone is active, clear the old pin (`DR_CLEAR(tone_base)=tone_mask`). Set `tone_pin=pin`, `tone_base=p->gpio`, `tone_mask=1<<p->bit`; `pinMode(pin, OUTPUT)`; drive low. `tone_toggle_count=count`; `tone_timer.begin(tone_interrupt, usec);` (re-begin = simple restart on same-pin change). `__enable_irq();`
- **`tone_interrupt()`** (ISR): if `tone_toggle_count == 0` → `DR_TOGGLE`, return (infinite). Else `if (--tone_toggle_count)` → `DR_TOGGLE`; else → `DR_CLEAR` (end low), `tone_timer.end()`, `tone_pin=255` (auto-stop).
- **`noTone(uint8_t pin)`**: `if (pin >= CORE_NUM_DIGITAL) return;` `__disable_irq();` if `pin == tone_pin` → `tone_timer.end()`, `DR_CLEAR` (low), `tone_pin=255`. `__enable_irq();`

## Verification

- **QEMU gate (`tone_test`, `-icount`)** — greps `TONE=PASS`:
  1. **frequency** — `tone(pin, 1000)`; poll `digitalRead(pin)` in a `millis()`-bounded window, count level changes ⇒ ~`2·1000·window_s` toggles (deterministic under `-icount`). Confirms the pin runs at `freq`.
  2. **duration auto-stop** — `tone(pin, 1000, 100)`; after ~100 ms the pin stops changing (sample before/after a `delay`).
  3. **noTone** — `noTone(pin)` stops toggling immediately.
  4. **frequency change** — `tone(pin, 2000)` after a `1000` tone ⇒ toggle rate doubles.
- **Saleae:** `tone(D9, 1000)` → measure **1000 Hz** square wave; verify a `duration` burst length. Reference `evkb/pwm_test/measure_pwm.py`.
- **HW:** piezo/speaker on the tone pin (audible) or scope; VCOM status. RC-oscillator tolerance (~0.6%, same source as IntervalTimer) applies — tone pitch is RC-bounded.

## Error handling

- `tone`/`noTone` bounds-check `pin` and ignore unmapped pins (`gpio==0`).
- `freq` clamped to ≥ 1 Hz; very high `freq` yields a small `usec` — still ≥ IntervalTimer's ~18-tick minimum (65535 Hz ⇒ 7.6 µs ⇒ ~183 ticks, safe).
- `tone` on a pin already playing: restarts at the new freq/duration (no smooth transition).
- `tone` when all 4 PIT channels are busy: `IntervalTimer::begin` returns false ⇒ tone silently does not start (documented; user frees a timer). ISR runs in interrupt context — keep it to the toggle + count.

## Risks / open items (resolved in plan)

- Confirm `__disable_irq`/`__enable_irq` intrinsics are available to the core (CMSIS) — else use a `PRIMASK` save/restore.
- Confirm `digitalRead(pin)` reflects an **output** pin's driven level (GPIO `PSR`/`DR` read) in QEMU and HW — needed for the gate's edge count; check `digital.c`.
- Exact `duration` toggle count / odd-rounding (whole waveform cycles, end low) matched against `cores/teensy4/tone.cpp`.
- `tone_timer.begin()` re-invoked on an already-running `tone_timer` (same-pin restart) reuses its channel cleanly — verify against `IntervalTimer::beginPeriod` (it reuses `channel` when already allocated).
